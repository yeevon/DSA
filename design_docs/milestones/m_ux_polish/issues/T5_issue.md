## 🟢 LOW — flag-only

### M-UX-T5-ISS-01 / LOW — `data-chapter-id` duplicated on `<article>` + inner `<span class="chapter-card-badge">` (intentional; informational only)

**Observation.** Each chapter card's `<article>` element carries `data-chapter-id={id}` (ChapterCard.astro line 55) AND its inner `<span class="chapter-card-badge">` also carries `data-chapter-id={id}` (line 63). Auditor's raw `grep -oE 'data-chapter-id="ch_[0-9]+"' dist/client/index.html \| wc -l` returns 24 hits (12 chapters × 2 carriers each), not 12.

**Why not a finding.** The duplication is intentional + a forward-compat hook for M5: the inner `.chapter-card-badge` is the M5 paint target for "X of Y sections read" completion glyphs. M5's island will query badge nodes by `[data-chapter-id="ch_N"]` directly (consistent with T2's `CompletionIndicator.astro` selector pattern that queries `.checkmark-slot[data-chapter-id]` — same idiom). Without the attribute on the inner span, M5 would have to traverse from the outer `<article>` down — an avoidable selector-traversal step and a forward asymmetry vs T2's `.checkmark-slot`. The Builder's choice mirrors T2's pattern for consistency.

**Action / Recommendation.** No action. Flag-only so a future audit doesn't report "duplicated `data-chapter-id` smells" without context. ### M-UX-T5-ISS-02 / LOW — `<h3>` heading construction places visually-hidden full title BEFORE the visible subtitle (accessibility nuance, not a bug)

**Observation.** ChapterCard.astro line 58–60 renders the heading as:
```astro
<h3 class="chapter-card-title">
  <span class="visually-hidden">{title} — </span>{subtitle}
</h3>
```
Sighted users see only the subtitle (e.g. "Lists, stacks, queues, deques"); screen readers hear "Chapter 4 — Lists, stacks, queues, deques" because the `.visually-hidden` span renders to assistive tech but is positioned-off-screen visually. This satisfies T5 spec line 61 ("Each card is a `<article>` with an `<h3>` heading (chapter number + title)").

**Why not a finding.** The pattern is correct AT spec level — sighted users get the chapter-number affordance via the prominent `<span class="chapter-card-num">ch_4</span>` tag in the header (a separate sibling element styled with the accent colour). The screen-reader text adds context that the visual `ch_4` tag carries iconically. The `<h3>` is well-formed (text content concatenates the visually-hidden prefix + the subtitle naturally). The inline `aria-hidden="true"` on the `chapter-card-num` span (line 57) prevents the screen reader from also announcing "ch underscore 4" — only the `<h3>` text reaches assistive tech.

**Action / Recommendation.** No action. Flag-only so a future accessibility audit doesn't "fix" the visually-hidden span by removing it (which would drop the chapter-number context for screen-reader users). The pattern matches T2's `LeftRail.astro` chapter-link `<span aria-hidden="true">ch_{n}</span>` + visible `{subtitle}` idiom (cross-component consistency). **Owner:** None — informational only; T7's responsive sweep + T8's deploy verification can validate this works under VoiceOver/NVDA/JAWS if the user wants live a11y verification.
