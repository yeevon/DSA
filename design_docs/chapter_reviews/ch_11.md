## Chapter map

**Opening `ideabox` (lines 11–13)** — sits before §11.1 as an orienting "Why B-trees exist" frame. Verbatim:

> **Why B-trees exist.** AVL and red-black trees (ch.\ 9) optimize for in-memory comparisons: each node holds one key and two children. But when your tree doesn't fit in RAM -- when it lives on disk or across a network -- a binary tree is a disaster. Each node traversal is a separate I/O operation. A tree of a billion keys is $\log_2 10^9 \approx 30$ levels deep; that's 30 disk seeks per lookup at roughly 10ms each = 300ms per query. A B-tree of the same size is $\log_{100} 10^9 \approx 4.5$ levels deep. Fatter nodes, fewer levels, fewer I/Os. *B-trees are what happens when you let cache lines or disk blocks dictate your node size.*

**Top gotchas (5 listed):**
1. Using the wrong "order" definition — "order $m$" means $m$ children max, $m-1$ keys max. Stay consistent.
2. Forgetting the root is the only node allowed to have $< \lceil m/2 \rceil - 1$ keys.
3. Splitting post-insertion instead of preemptively — doable but forces a second pass.
4. Remove without rotating/fusing first — leaves a node with too few keys; violates the invariant.
5. Using B-tree where a hash index would do. B-trees earn their keep on range queries and sorted iteration.
